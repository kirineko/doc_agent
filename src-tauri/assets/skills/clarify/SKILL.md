# Clarify Skill — 文档需求澄清

通过 `clarify_ask` 工具逐问澄清文档的内容、结构与排版样式，输出结构化「创作简报」，再进入对应格式的生成或编辑流程。

适用于 Word / PPT / Excel / HTML 报告等所有文档类型，覆盖**全新创作**与**编辑已有文件**两类场景。

## 何时使用

遇到以下任一情况时，使用本流程进行澄清：

- **全新创作**：用户要求生成新文档，但缺少以下 **≥ 2 项**信息：主题/内容大纲、目标受众、文档结构、排版风格偏好、篇幅要求
- **编辑已有文件**：用户要求修改/完善文档，但改动目标不够明确（如「帮我优化一下」「感觉哪里不对劲」「排版有点乱」等）

**不需要澄清**：用户已给出明确单点指令（「把第二页标题改成蓝色」「在第三章后面加一节」），直接执行即可。用户明确说「你看着办 / 按默认来」时，给出推荐后一次确认即可，无需完整澄清流程。

## 执行流程

1. **识别文档类型** — 若未说明（Word / PPT / Excel / 报告），作为第一个问题
2. **识别场景** — 全新创作 or 编辑已有文件
3. **评估已有信息** — 对照下方关键信息清单，标记已明确项与缺口
4. **选择深度路径**（见下节）
5. **逐问澄清** — **每轮只问一个问题**；每问 MUST 调用 `clarify_ask`；用户已给出的信息跳过
6. **汇总确认** — 用 `clarify_ask` 且 `kind=confirm_brief` 展示「创作简报」，**等待用户明确确认后**再进入生成
7. **转入生成** — `skill_read` 对应格式 skill，按创作简报执行

澄清过程中**不得**调用 `skill_run` 或直接写交付物；不得只用 assistant 纯文本列出选项或问卷。

## clarify_ask 使用规则

每一道澄清问题 MUST 使用 `clarify_ask`，不要直接在正文里要求用户回答 A/B/C。

常用题型：

- `single`：文档类型、场景、风格、受众等单选问题
- `multi`：章节、必须包含的内容、图表需求等多选问题
- `text`：主题、目的、自由描述
- `confirm_brief`：创作简报确认（澄清结束必用）

示例：

```json
{
  "tool": "clarify_ask",
  "args": {
    "id": "ppt_style",
    "kind": "single",
    "prompt": "这份 PPT 的视觉风格您更倾向哪种？",
    "options": [
      { "id": "minimal", "label": "简约留白" },
      { "id": "business_dark", "label": "商务深色" },
      { "id": "visual", "label": "图文并茂" }
    ],
    "allow_custom": true
  }
}
```

## 深度控制

根据用户请求中**已包含**的关键信息数量选择路径：

| 路径 | 条件 | 问题数 |
|------|------|--------|
| 极简 | 已含 ≥ 3 项关键信息 | 1–2 个（仅补缺口或确认） |
| 标准（默认） | 已含 1–2 项 | 4–6 个 |
| 深度 | 极为模糊（≤ 1 项） | 最多 8 个 |

**覆盖要求**：标准与深度路径中，**至少问 1 个排版/样式相关问题**（除非用户已明确风格/配色/字体）。

## 关键信息检查清单

澄清时对照此清单，跳过已有答案的项：

- [ ] 文档类型（Word / PPT / Excel / 报告）
- [ ] 主题与核心目的 / 编辑目标
- [ ] 目标受众 / 使用场景
- [ ] 结构与篇幅 / 改动范围
- [ ] 排版 / 样式 / 视觉风格

## 问题库 — Word 文档（docx）

按缺口从下列维度选题，**每次只问一题**。

### 内容

- 这份文档的主题和核心目的是什么？（编辑场景：希望达到什么效果？）
- 读者是谁？（内部 / 外部、职级、是否熟悉专业术语）
- 有哪些必须覆盖的关键论点或数据？

### 结构

- 预期有哪些章节？是否需要摘要、目录、附录？
- 篇幅大约多少页或多少字？
- 是否需要封面、页眉页脚、页码？

### 排版 / 样式

- 整体风格倾向？（公文正式 / 商务报告 / 学术 / 现代简洁）
- 正文字体偏好？（宋体 / 微软雅黑 / 等线 / 无要求）
- 行距与段距？（宽松阅读 / 紧凑商务）
- 标题层级用编号（1. 1.1 1.1.1）还是纯样式区分？
- 是否有企业模板、品牌色或 logo 要求？

---

## 问题库 — PPT 演示（pptx）

### 内容

- 演讲主题和希望听众记住的核心结论是什么？（编辑场景：希望调整哪方面？）
- 听众与场景？（内部团队 / 客户 / 管理层 / 公众）
- 必须出现的数据、案例或关键信息有哪些？

### 结构

- 大约多少张幻灯片，或演讲时长？
- 叙事结构？（问题-方案-行动 / 时间轴 / 对比分析 / 其他）
- 是否需要封面、目录、致谢或联系页？

### 排版 / 样式

- 视觉风格？（简约留白 / 商务深色 / 图文并茂 / 数据驱动）
- 主色调或品牌色 / logo 要求？
- 文字密度？（每页一句结论 / 可接受较多文字）
- 是否需要统一动画？

---

## 问题库 — Excel 表格（xlsx）

### 内容

- 表格用途？（数据录入 / 分析计算 / 展示报表 / 仪表盘）
- 数据来源与核心字段有哪些？（编辑场景：需要新增/修改哪些字段或公式？）
- 是否有计算逻辑或公式需求？（汇总 / 环比 / 条件统计等）

### 结构

- 大约几张工作表？各自的用途？
- 是否需要数据验证、下拉菜单或保护某些区域？
- 数据量级？（行数估计，影响性能与样式设计）

### 排版 / 样式

- 整体风格？（简洁数据型 / 正式报表 / 带品牌色彩的展示型）
- 表头与合并单元格的偏好？
- 是否需要条件格式（热力图、颜色标注超标值等）？
- 图表需求？（类型、数量，是否嵌入工作表）
- 是否有企业模板或配色要求？

---

## 问题库 — 报告 / HTML Report

适用于静态 HTML 报告或长篇书面报告。

### 内容

- 报告类型？（数据分析 / 项目总结 / 研究报告 / 调研报告）
- 数据来源与核心指标是什么？
- 结论导向？（描述现状 / 诊断问题 / 提出建议）

### 结构

- 必须包含哪些章节？（摘要 / 背景 / 分析 / 结论 / 建议）
- 需要哪些图表？类型、数量、数据是否已有？
- 主要阅读场景？（屏幕 / 打印 / 在线分享）与大致篇幅？

### 排版 / 样式

- 配色方案？（企业品牌色 / 中性灰蓝 / 自定义）
- 是否需要植入 logo 或遵循品牌规范？
- 图表样式？（简约线条 / 填充色块 / 渐变）
- 是否需要打印友好布局（页宽、分页、黑白可读）？

## 创作简报（澄清结束输出）

澄清完成后**必须**用 `clarify_ask` 的 `confirm_brief` 题型提交以下字段，等待用户明确确认后再进入生成。

`brief` MUST 为**扁平**对象：字段名直接作为 key，值为字符串。禁止把全部字段再包一层 `{ "创作简报": { ... } }`。

```json
{
  "id": "creation_brief",
  "kind": "confirm_brief",
  "prompt": "以下是本次 PPT 的【创作简报】，请确认是否按此方向执行？",
  "brief": {
    "文档类型": "PPT",
    "主题/目标": "广软人工智能产业学院建设方案汇报",
    "受众/场景": "董事会 / 校领导",
    "结构": "封面、目录、背景、必要性、目标、内容、合作、财务、结语（8–12 页）",
    "排版风格": "简约留白，内容优先",
    "样式要点": "正文清晰，数据用表格或图表呈现",
    "特殊要求": "突出 ROI 与预期营收"
  }
}
```

字段参考：

```
文档类型：<Word / PPT / Excel / 报告>
主题 / 目标：<一句话>
受众 / 使用场景：<描述>
结构：<章节、幻灯片或工作表框架>
排版风格：<整体视觉/文档风格>
样式要点：<字体、配色、间距、图表样式等>
特殊要求：<其他约束，无则写「无」>
```

用户确认后执行：

1. `skill_read` 对应格式（`docx` / `pptx` / `xlsx` / `html-report`）
2. 按创作简报生成或编辑交付物

## 提问原则

- **一次一问**，不要列出问卷
- **多选题优先**（给出 2–4 个选项 + 「其他」）
- 用户已说过的内容**不要重复问**，可简短确认
- 用户说「你看着办」时，给出你的推荐并请求一次确认即可
