# Clarify Skill — 文档需求澄清

通过 `clarify_ask` 工具逐问澄清文档的内容、结构与排版样式，输出结构化「创作简报」，再进入对应格式的生成或编辑流程。

适用于 Word / PPT / Excel / HTML 报告 / Typst PDF 等交付形态，覆盖**全新创作**与**编辑已有文件**两类场景。

## 何时使用

遇到以下任一情况时，使用本流程进行澄清：

- **全新创作**：用户要求生成新文档，但缺少以下 **≥ 2 项**信息：主题/内容大纲、目标受众、文档结构、排版风格偏好、篇幅要求
- **编辑已有文件**：用户要求修改/完善文档，但改动目标不够明确（如「帮我优化一下」「感觉哪里不对劲」「排版有点乱」等）

**不需要澄清**：用户已给出明确单点指令（「把第二页标题改成蓝色」「在第三章后面加一节」），直接执行即可。用户明确说「你看着办 / 按默认来」时，给出推荐后一次确认即可，无需完整澄清流程。

## 执行流程

0. **读取项目配置** — 若 system 已含 `## 项目配置（AGENTS.md）`，将其视为**默认创作规范**；若无该段，可 `fs_read AGENTS.md`（`exists:false` 则跳过）。详见下节「与 AGENTS.md 协作」
1. **识别交付格式** — 若未说明（Word / PPT / Excel / HTML 报告 / Typst PDF），作为第一个或第二个问题；**AGENTS.md 已指定默认交付类型时跳过**
2. **识别场景** — 全新创作 or 编辑已有文件
3. **评估已有信息** — 对照下方关键信息清单 **与 AGENTS.md 已覆盖项**，标记已明确项与**本次任务**仍缺的缺口
4. **选择深度路径**（见下节；有 AGENTS.md 时倾向极简）
5. **逐问澄清** — **每轮只问一个问题**；每问 MUST 调用 `clarify_ask`；用户已给出的信息 **及 AGENTS.md 已规定项** 跳过
6. **汇总确认** — 用 `clarify_ask` 且 `kind=confirm_brief` 展示「创作简报」，**等待用户明确确认后**再进入生成
7. **转入生成** — 按创作简报中的「交付格式」走对应分支（见下节「澄清后执行」）

澄清过程中**不得**调用 `skill_run`、`typst_to_pdf` 或直接写最终交付物；不得只用 assistant 纯文本列出选项或问卷。

## 与 AGENTS.md 协作

项目根 `AGENTS.md` 描述**跨任务**的默认偏好（各 Office 类型的风格、命名、参考文件、禁止事项等）。文档创作 clarify **不是** `/init`，但 MUST 尊重已有配置，避免重复问卷。

### 何时视为「已有配置」

- system 消息含 `## 项目配置（AGENTS.md）` 段（每 turn 自动注入），或
- `fs_read AGENTS.md` 返回 `exists: true`

### 继承规则（MUST）

| AGENTS.md 中已写明 | clarify 行为 |
|-------------------|--------------|
| 某 Office 类型的风格 / 配色 / 字体 / 页数习惯 | **不问**同类排版/样式题；写入简报「排版风格」「样式要点」时**引用或概括 AGENTS.md** |
| 命名与路径、参考文件、禁止事项 | **不问**；执行阶段遵守，简报「参考材料」「特殊要求」可写「见 AGENTS.md / @路径」 |
| 概述里写了项目主交付物类型 | 用户未指定格式时**优先采用**，仅当本次任务明显不符时再问交付格式 |
| 某章节缺失或与本次交付类型无关 | 仅问**本次任务**相关且 AGENTS.md **未覆盖**的缺口 |

### 仍应澄清的内容（即使 AGENTS.md 存在）

- **本次**文档的主题、目的、核心内容、改动范围（编辑场景）
- **本次**结构与篇幅（章节、页数、幻灯片数——除非 AGENTS.md 对**该类型**已有明确默认且用户未提出新目标）
- 用户消息与 AGENTS.md **冲突**时：只问 **1 题** 确认以哪方为准
- 用户明确要求「这次例外 / 换风格」时：只问与例外相关的最小缺口

> 深度路径如何受 AGENTS.md 影响，见下文「深度控制」表（已纳入 AGENTS.md 覆盖度）。

### confirm_brief 写法

- 「排版风格」「样式要点」：优先摘录 AGENTS.md 相关句，后缀注明「（沿用项目 AGENTS.md）」
- 仅对**本次任务特有**且 AGENTS.md 未写的内容单独填字段
- 勿在简报中重复粘贴 AGENTS.md 全文

## clarify_ask 使用规则

每一道澄清问题 MUST 使用 `clarify_ask`，不要直接在正文里要求用户回答 A/B/C。

常用题型：

- `single`：交付格式、场景、风格、受众等单选问题
- `multi`：章节、必须包含的内容、图表需求等多选问题
- `text`：主题、目的、自由描述
- `confirm_brief`：创作简报确认（澄清结束必用）

示例：

```json
{
  "tool": "clarify_ask",
  "args": {
    "id": "ppt_style",
    "kind": "single",
    "prompt": "这份 PPT 的视觉风格您更倾向哪种？",
    "options": [
      { "id": "minimal", "label": "简约留白" },
      { "id": "business_dark", "label": "商务深色" },
      { "id": "visual", "label": "图文并茂" }
    ],
    "allow_custom": true
  }
}
```

## 深度控制

根据用户请求中**已包含**的关键信息数量 **及 AGENTS.md 已覆盖项** 选择路径：

| 路径 | 条件 | 问题数 |
|------|------|--------|
| 极简 | 已含 ≥ 3 项关键信息，**或** AGENTS.md 已覆盖本次类型的样式/规范且任务意图清晰 | 0–2 个（常直接 confirm_brief） |
| 标准（默认） | 已含 1–2 项，或 AGENTS.md 部分覆盖 | 2–4 个（跳过 AGENTS.md 已有项） |
| 深度 | 极为模糊（≤ 1 项）且 AGENTS.md 无帮助 | 最多 8 个 |

**覆盖要求**：标准与深度路径中，**至少问 1 个排版/样式相关问题**——**除非**用户已明确风格/配色/字体，**或** AGENTS.md 已对本次交付类型规定样式（此时跳过，在 confirm_brief 中引用即可）。

## 关键信息检查清单

澄清时对照此清单，跳过已有答案的项 **及 AGENTS.md 已规定项**：

- [ ] 交付格式（Word / PPT / Excel / HTML 报告 / Typst PDF）— AGENTS.md 有默认且与任务一致则跳过
- [ ] 主题与核心目的 / 编辑目标 — **几乎总是要问**（任务特有）
- [ ] 目标受众 / 使用场景 — AGENTS.md「概述」已写且用户未改意图则跳过
- [ ] 结构与篇幅 / 改动范围 — **任务特有**；AGENTS.md 仅有类型级默认（如「PPT 8–12 页」）且用户未提新目标时可跳过
- [ ] 排版 / 样式 / 视觉风格 — **AGENTS.md 对应章节已写则跳过**
- [ ] 参考材料（PDF / 图片 / 已有项目文件路径，无则跳过）— AGENTS.md「参考文件」已列则跳过
- [ ] 公式或版式要求（公式多 → 默认建议 Typst PDF；坚持 Word → 简报中写明）— AGENTS.md 已规定则跳过

## 交付格式与参考材料（通用）

**交付格式**（用户未说明时单选一问）：

- Word（.docx）
- PPT（.pptx）
- Excel（.xlsx）
- HTML 报告（可后续 `html_to_pdf`）
- Typst PDF（公式密集、版式严谨时优先推荐）
- 仅项目内中间稿（.md / .typ，无最终打包文件）

**参考材料**：用户提到旧 PDF、扫描件、题图时，简报中记录 `@` 路径；生成前用 `pdf_read`（默认，vision 模型）或 `image_read` 读取，**勿凭记忆复述**内容。

**公式分工**（写入简报「特殊要求」或口头确认即可）：

- 公式多、版式严 → 建议 **Typst PDF**
- 必须 Word 且含公式 → 交付格式定为 Word，生成时 `skill_read docx` 后再读 `math.md`
- 图文网页型报告 → HTML 报告 + 可选 `html_to_pdf`

## 问题库 — Word 文档（docx）

按缺口从下列维度选题，**每次只问一题**。**AGENTS.md「Word」章节已写明的排版/样式/结构习惯一律跳过。**

### 内容

- 这份文档的主题和核心目的是什么？（编辑场景：希望达到什么效果？）
- 读者是谁？（内部 / 外部、职级、是否熟悉专业术语）
- 有哪些必须覆盖的关键论点或数据？

### 结构

- 预期有哪些章节？是否需要摘要、目录、附录？
- 篇幅大约多少页或多少字？
- 是否需要封面、页眉页脚、页码？

### 排版 / 样式

- 整体风格倾向？（公文正式 / 商务报告 / 学术 / 现代简洁）
- 正文字体偏好？（宋体 / 微软雅黑 / 等线 / 无要求）
- 行距与段距？（宽松阅读 / 紧凑商务）
- 标题层级用编号（1. 1.1 1.1.1）还是纯样式区分？
- 是否有企业模板、品牌色或 logo 要求？

---

## 问题库 — PPT 演示（pptx）

**AGENTS.md「PPT」章节已写明的视觉风格、页数结构等一律跳过。**

### 内容

- 演讲主题和希望听众记住的核心结论是什么？（编辑场景：希望调整哪方面？）
- 听众与场景？（内部团队 / 客户 / 管理层 / 公众）
- 必须出现的数据、案例或关键信息有哪些？

### 结构

- 大约多少张幻灯片，或演讲时长？
- 叙事结构？（问题-方案-行动 / 时间轴 / 对比分析 / 其他）
- 是否需要封面、目录、致谢或联系页？

### 排版 / 样式

- 视觉风格？（简约留白 / 商务深色 / 图文并茂 / 数据驱动）
- 主色调或品牌色 / logo 要求？
- 文字密度？（每页一句结论 / 可接受较多文字）
- 是否需要统一动画？

## 问题库 — Excel 表格（xlsx）

**AGENTS.md「Excel」章节已写明的表头/样式习惯一律跳过。**

### 内容

- 表格用途？（数据录入 / 分析计算 / 展示报表 / 仪表盘）
- 数据来源与核心字段有哪些？（编辑场景：需要新增/修改哪些字段或公式？）
- 是否有计算逻辑或公式需求？（汇总 / 环比 / 条件统计等）

### 结构

- 大约几张工作表？各自的用途？
- 是否需要数据验证、下拉菜单或保护某些区域？
- 数据量级？（行数估计，影响性能与样式设计）

### 排版 / 样式

- 整体风格？（简洁数据型 / 正式报表 / 带品牌色彩的展示型）
- 表头与合并单元格的偏好？
- 是否需要条件格式（热力图、颜色标注超标值等）？
- 图表需求？（类型、数量，是否嵌入工作表）
- 是否有企业模板或配色要求？

## 问题库 — 报告 / HTML Report

适用于静态 HTML 报告或长篇书面报告。

### 内容

- 报告类型？（数据分析 / 项目总结 / 研究报告 / 调研报告）
- 数据来源与核心指标是什么？
- 结论导向？（描述现状 / 诊断问题 / 提出建议）

### 结构

- 必须包含哪些章节？（摘要 / 背景 / 分析 / 结论 / 建议）
- 需要哪些图表？类型、数量、数据是否已有？
- 主要阅读场景？（屏幕 / 打印 / 在线分享）与大致篇幅？

### 排版 / 样式

- 配色方案？（企业品牌色 / 中性灰蓝 / 自定义）
- 是否需要植入 logo 或遵循品牌规范？
- 图表样式？（简约线条 / 填充色块 / 渐变）
- 是否需要打印友好布局（页宽、分页、黑白可读）？

## 创作简报（澄清结束输出）

澄清完成后**必须**用 `clarify_ask` 的 `confirm_brief` 题型提交以下字段，等待用户明确确认后再进入生成。

`brief` MUST 为**扁平**对象：字段名直接作为 key，值为字符串。禁止把全部字段再包一层 `{ "创作简报": { ... } }`。

```json
{
  "id": "creation_brief",
  "kind": "confirm_brief",
  "prompt": "以下是本次 PPT 的【创作简报】，请确认是否按此方向执行？",
  "brief": {
    "交付格式": "PPT",
    "主题/目标": "广软人工智能产业学院建设方案汇报",
    "受众/场景": "董事会 / 校领导",
    "结构": "封面、目录、背景、必要性、目标、内容、合作、财务、结语（8–12 页）",
    "排版风格": "简约留白，内容优先",
    "样式要点": "正文清晰，数据用表格或图表呈现",
    "参考材料": "无",
    "特殊要求": "突出 ROI 与预期营收"
  }
}
```

字段参考：

```
交付格式：<Word / PPT / Excel / HTML 报告 / Typst PDF / 中间稿>
主题 / 目标：<一句话>
受众 / 使用场景：<描述>
结构：<章节、幻灯片或工作表框架>
排版风格：<整体视觉/文档风格>
样式要点：<字体、配色、间距、图表样式等>
参考材料：<项目内路径或「无」>
特殊要求：<公式、页数、是否基于参考材料改编等；无则写「无」>
```

## 澄清后执行（按交付格式）

用户确认创作简报后，**仅**走对应分支：

| 交付格式 | 执行顺序 |
|----------|----------|
| **Word** | `skill_read docx` → 含公式再 `skill_read docx/math.md` → `skill_run` 或 `ooxml_unpack` 编辑链 |
| **PPT** | `skill_read pptx`（必要时 `pptxgenjs.md` / `editing.md`）→ `skill_run` |
| **Excel** | `skill_read xlsx` → `skill_run` / `data_query` 等 |
| **HTML 报告** | `skill_read html-report` → 生成 HTML → 可选 `html_to_pdf` |
| **Typst PDF** | ① `typst_read_template syntax/typst-guide` → ② `typst_list_templates` → ③ `typst_read_template` 选场景模板 → ④ `fs_write` 项目内 `.typ` → ⑤ `typst_to_pdf`（仅重编译已有 `.typ` 可跳过 ②③） |
| **中间稿** | `fs_write` 草稿（如 `.md` / `.typ`），待用户确认后再定最终格式 |

有参考 PDF/图片时，在以上任一分支之前或之中先完成 `pdf_read` / `image_read`。

用户确认后**不要**再跑完整 clarify；按简报一项技能读入规范即可开始生成。

## 提问原则

- **一次一问**，不要列出问卷；**每轮 assistant 仅一个 clarify_ask**
- **先读 AGENTS.md（system 段或 fs_read）**，已规定的项目级样式/规范**不要重复问**
- 只问**本次任务**仍不清楚的内容（主题、具体结构、编辑范围、与 AGENTS.md 的冲突）
- **多选题优先**（给出 **2–8 个选项**，用 `allow_custom: true` 承接「其他」；`options` 总数不得超过 12）
- 用户已说过的内容**不要重复问**，可简短确认
- 用户说「你看着办」时：有 AGENTS.md 则按项目默认 + 任务上下文直接 `confirm_brief`；无则给出推荐并请求一次确认
