## Context

doc-agent 当前在收到文档创作请求时会直接进入 `skill_read` → `skill_run` 流程，没有需求澄清环节。当用户描述模糊（仅说"写一份报告"或"做个 PPT"）时，agent 只能猜测关键参数（受众、结构、风格等），生成结果与预期偏差大。

现有 skill 体系（`docx`/`pdf`/`pptx`/`xlsx`/`html-report`）均为"如何生成"的执行指南，没有"生成什么"的澄清指导。新增 `clarify` skill 填补这一缺口，与现有 skill 架构完全对齐。

## Goals / Non-Goals

**Goals:**
- 新增 `clarify` skill，作为文档创作前的需求澄清指南
- 覆盖 Word / PPT / 报告三类文档的内容、结构、排版/样式问题库
- 支持自适应深度：根据用户初始请求的完整程度决定问题数量
- 澄清流程输出结构化「创作简报」，直接驱动后续 `skill_run`
- 在系统提示词中注入触发条件，使 agent 在模糊请求时自动执行澄清

**Non-Goals:**
- 不集成 Visual Companion（本地 HTTP server + 浏览器交互）
- 不为 PDF 表单、xlsx 数据分析等非创作类任务提供澄清流程
- 不改变 skill_run 执行逻辑，clarify 只影响生成前的信息收集

## Decisions

### D1：以 embedded skill asset 形式集成

`clarify` skill 以 `src-tauri/assets/skills/clarify/SKILL.md` 形式编译进二进制，通过现有 `skill_read` 工具获取，与 `docx`/`pptx` 等 skill 完全一致。

**备选方案**：写进系统提示词 → 提示词过长，维护困难，且无法按需加载（渐进披露原则被破坏）。

### D2：触发条件写进系统提示词（而非 skill 本身）

触发逻辑（何时 `skill_read clarify`）写入 `loop_runner.rs` 的系统提示词，而非 skill 内容本身。这样 agent 在读取 skill 之前就知道是否需要读它，符合渐进披露的设计原则。

触发条件：
1. 请求意图是**全新文档创作**（非编辑/修改已有文件）
2. 缺少以下信息中的 ≥ 2 项：主题/内容大纲、目标受众、结构框架、风格偏好、篇幅
3. 请求规模不是单点操作（"改字体"、"加表格"等不触发）

### D3：clarify skill 包含三类文档的独立问题库

skill 文档按文档类型分组，每组含内容问题 + 结构问题 + 排版/样式问题。agent 根据用户实际文档类型选择对应问题库，已在初始描述中给出的信息对应的问题自动跳过。

### D4：自适应深度控制

| 路径 | 触发条件 | 问题数量 |
|------|---------|---------|
| 极简 | 初始描述已含 ≥ 3 项关键信息 | 1-2 个确认问题 |
| 标准（默认） | 初始描述含 1-2 项信息 | 4-6 个问题 |
| 深度 | 初始描述极为模糊（< 1 项明确信息） | 最多 8 个问题 |

每次只问一个问题，用户回答后继续下一个，而非一次列出所有问题。

### D5：创作简报格式

澄清完成后，agent 输出结构化创作简报，格式固定，便于后续 `skill_read` + `skill_run` 直接引用：

```
【创作简报】
文档类型：<类型>
主题：<主题>
受众：<受众描述>
结构：<章节/幻灯片列表>
排版风格：<风格描述>
样式要点：<字体/配色/间距等>
特殊要求：<其他约束>
```

## Risks / Trade-offs

- **风险**：agent 对触发条件判断不准确，在不需要澄清时触发（false positive）→ 缓解：触发条件措辞明确，且条件为 AND 逻辑，减少误触发
- **风险**：用户觉得澄清流程繁琐 → 缓解：极简路径存在，已有足够信息时只问 1-2 个确认问题
- **权衡**：skill 内容作为静态 Markdown，问题库无法动态扩展 → 接受，与现有 skill 体系一致，后续可通过修改 SKILL.md 扩展
